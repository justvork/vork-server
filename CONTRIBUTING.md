# Contributing to Vork

First of all, thank you.

Whether you've found a typo, spotted a bug, built a new Skill, or completely disagree with an architectural decision, you've already done something valuable: you've cared enough to get involved.

Vork exists because we believe software should evolve in public, shaped by the engineers who use it.

---

# The Philosophy

Vork is not trying to become another AI assistant.

It is an operating system for organisational behaviour.

AI is simply the natural language interface.

That distinction matters.

When contributing, ask yourself:

> **"Am I teaching Vork a new behaviour, or am I building another application?"**

We generally prefer small, composable behaviours over large monolithic features.

---

# Ways You Can Help

There are many ways to contribute.

## Report Bugs

If something doesn't work, tell us.

Please include:

- what you expected
- what happened
- relevant logs
- screenshots if appropriate
- steps to reproduce

The more information you provide, the easier it is to reproduce and fix.

---

## Improve Documentation

Great documentation is engineering.

If something confused you, it will probably confuse someone else.

Pull requests improving documentation are always welcome.

---

## Build Skills

One of the easiest ways to contribute is by creating reusable Skills.

Skills should aim to solve a single problem well.

Good examples include:

- GitHub automation
- Jira workflows
- CRM integration
- Certificate management
- Infrastructure deployment
- Documentation generation
- Customer support workflows

If you build something useful, please share it.

---

## Improve Existing Behaviours

Sometimes the best contribution isn't a new feature.

It's making an existing behaviour simpler, clearer or more reusable.

Always ask:

> Could someone else benefit from this abstraction?

---

## Improve the Platform

Core contributions are welcome.

Examples include:

- performance improvements
- security enhancements
- new persistence providers
- notification providers
- AI providers
- OAuth integrations
- tooling
- testing
- developer experience

---

# Before Opening a Pull Request

Please ask yourself:

- Is this behaviour generally useful?
- Does it simplify the platform?
- Does it reduce work?
- Does it preserve security?
- Would another organisation benefit from this?

Small focused pull requests are preferred over very large changes.

---

# Coding Style

Vork deliberately keeps dependencies to a minimum.

We generally prefer:

- simple Java
- clear APIs
- composition over complexity
- readable code
- explicit behaviour

If something needs a lengthy comment to explain it, consider whether it can be simplified instead.

---

# Security

Security is a core design principle.

Please do not submit changes that bypass:

- authorisation
- approval workflows
- credential boundaries
- secret storage
- auditability

If you discover a security issue, please report it privately rather than opening a public issue.

---

# Discuss Ideas First

Not every contribution needs to begin with code.

Some of Vork's biggest architectural ideas started as conversations.

If you're unsure whether something belongs in the platform, open a discussion first.

We'd much rather discuss an idea early than review thousands of lines of code heading in the wrong direction.

---

# The Workshop

Many of the ideas behind Vork are explored publicly before they become software.

Founder Lee Painter documents observations, experiments and engineering principles in **The Workshop**.

If you're interested in understanding *why* Vork is built the way it is, it's a good place to start.

https://leepainter.com

---

# Our Promise

We will always try to review contributions respectfully.

Sometimes we'll say no.

Usually because we're trying to keep the platform simple rather than because the contribution lacks merit.

If we reject a proposal, we'll explain why.

Good ideas often come back later in a different form.

---

# Show Us What You Built

Vork isn't trying to replace developers.

It's trying to change what developers build.

Every behaviour shared with the community helps someone else solve a problem they haven't encountered yet.

Whether you contribute code, documentation, a new Skill, or simply share your experience on LinkedIn, you're helping move the project forward.

If you post about Vork, tell us two things:

- **What did you build?**
- **What software did it replace?**

Those stories matter.

They teach.

They inspire.

They challenge assumptions.

And sometimes they become the next great behaviour.

We can't wait to see what you build.
